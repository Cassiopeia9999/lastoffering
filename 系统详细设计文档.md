# 校园失物招领智能管理系统详细设计文档

## 1. 引言

### 1.1 编写目的

本文档用于对校园失物招领智能管理系统的体系结构、功能模块、数据库、接口、关键业务流程、AI 能力及部署方案进行详细说明，为系统的开发实现、测试验证、论文撰写、答辩展示和后续维护提供统一依据。

### 1.2 文档依据

本文档主要依据以下内容编写：

- 当前项目源代码与现有页面实现
- 项目现有运行结构与部署配置
- 需求参考文档：[39-2022132019-刘埌铖-需求规格说明书.docx](F:/Last_offering/items/docs/39-2022132019-刘埌铖-需求规格说明书.docx)

特别说明：

- 本详细设计文档以当前系统实际实现为准。
- 需求规格说明书主要作为功能划分、术语和业务目标的参考依据。
- 若需求规格说明书与当前系统存在差异，应以当前代码、数据库模型和页面行为为准。

### 1.3 系统概述

本系统面向高校校园失物招领场景，采用前后端分离架构实现失物与招领信息管理、智能图像识别与以图搜物、自然语言智能寻物、快速发布、留言通知、匹配确认和管理员后台管理等能力。

系统在实现层面兼顾了传统 Web 信息管理功能与智能增强能力，既满足基础的发布查询需求，又通过图像特征检索、语义解析和异步匹配提升了失物招领效率。

### 1.4 术语定义

| 术语 | 说明 |
|---|---|
| 失物 | 用户发布的遗失记录，`type=lost` |
| 招领 | 用户发布的拾得记录，`type=found` |
| 匹配记录 | 系统对一条失物记录与一条招领记录建立的认领关系 |
| 特征向量 | 图像特征提取模型输出的向量表示，用于相似度计算 |
| 语义归一化 | 对类别、颜色、品牌、地点和关键词进行规范化处理 |
| 快速发布 | 根据自然语言自动生成物品发布字段的功能 |
| 智能寻物 | 根据自然语言提取检索条件并返回排序结果的功能 |

## 2. 总体设计

### 2.1 设计目标

系统详细设计围绕以下目标展开：

1. 构建统一的校园失物招领信息管理平台。
2. 支持失物与招领信息的完整生命周期管理。
3. 利用图像特征提取与相似度检索提升查找效率。
4. 引入自然语言解析能力降低使用门槛。
5. 形成从发布、检索、沟通、匹配到认领完成的业务闭环。
6. 为管理员提供平台治理和运行统计能力。

### 2.2 总体架构

系统采用典型的前后端分离架构，由表现层、业务层、数据层和智能服务层组成。

#### 2.2.1 表现层

表现层由 Vue 3 前端实现，负责页面展示、表单交互、状态管理、消息提示和路由跳转。普通用户端与管理员端共用前端工程，通过不同布局和路由实现页面隔离。

#### 2.2.2 业务层

业务层由 FastAPI 后端实现，负责用户认证、物品管理、匹配流转、通知生成、留言处理、后台管理以及 AI 服务调度。

#### 2.2.3 数据层

数据层使用 MySQL 存储结构化业务数据，上传图片保存在本地媒体目录。后端通过 SQLAlchemy ORM 对数据库实体进行映射和访问。

#### 2.2.4 智能服务层

智能服务层由本地图像分类与特征提取模块、图像检索模块、自动匹配模块、语义规则模块以及 ARK 大模型增强模块组成。

### 2.3 技术选型

| 层次 | 技术方案 |
|---|---|
| 前端 | Vue 3、Vite、Vue Router、Pinia、Element Plus、Axios |
| 后端 | FastAPI、Uvicorn、Pydantic、SQLAlchemy |
| 数据库 | MySQL 8、PyMySQL |
| 认证 | JWT、OAuth2PasswordBearer、python-jose |
| 图像处理 | Pillow、Cropper.js |
| 图像智能 | PyTorch、本地图像分类与特征提取模型 |
| 语义智能 | 本地语义规则 + ARK 大模型接口 |
| 部署 | Docker Compose、Nginx |

### 2.4 功能模块划分

参考需求规格说明书中的 SYS1 至 SYS5 划分方式，并结合当前实现，系统功能模块划分如下：

| 模块编号 | 模块名称 | 当前实现说明 |
|---|---|---|

| SYS2 | 物品信息管理模块 | 已实现发布、编辑、查询、上下架、关闭、相似推荐 |
| SYS3 | 图像识别与以图搜物模块 | 已实现分类识别、特征提取、相似检索、异步自动匹配 |
| SYS4 | 消息与通知模块 | 已实现站内通知、留言、匹配确认反馈、完成认领提醒 |

## 3. 功能模块详细设计

### 3.1 SYS1 用户与数据管理模块

#### 3.1.1 模块职责

该模块负责用户注册、登录、身份鉴权、个人资料管理、头像上传及管理员权限判断，是其他业务模块运行的基础。

#### 3.1.2 主要实现文件

- [main.py](F:/Last_offering/items/backend/app/main.py)
- [user.py](F:/Last_offering/items/backend/app/api/v1/user.py)

#### 3.1.3 功能设计

系统提供如下核心功能：

- 用户注册
- 用户登录
- 获取当前登录用户信息
- 修改个人资料
- 上传或更换头像
- 区分普通用户与管理员权限

#### 3.1.4 认证与授权设计

系统采用 JWT 进行身份认证。登录成功后，后端签发访问令牌，前端在后续请求中通过 `Authorization: Bearer <token>` 方式提交。

权限控制依赖两类核心校验函数：

- 普通登录校验：`get_current_user`
- 管理员校验：`get_current_admin`

认证失败返回 401，权限不足返回 403，被禁用用户即使令牌有效也会被拒绝访问。

### 3.2 SYS2 物品信息管理模块

#### 3.2.1 模块职责

该模块负责失物与招领信息的创建、编辑、查询、状态更新、删除与恢复，并支撑后续匹配流程。

#### 3.2.2 主要实现文件

- [item.py](F:/Last_offering/items/backend/app/api/v1/item.py)
- [PublishView.vue](F:/Last_offering/items/frontend/src/views/PublishView.vue)
- [MyItemsView.vue](F:/Last_offering/items/frontend/src/views/MyItemsView.vue)
- [ItemDetailView.vue](F:/Last_offering/items/frontend/src/views/ItemDetailView.vue)

#### 3.2.3 关键字段设计

物品记录主要包含以下字段：

- 基础描述字段：`type`、`title`、`description`
- 结构化属性字段：`category`、`color`、`brand`、`keywords`、`feature_text`
- 位置信息字段：`location`、`happen_time`
- 图像相关字段：`image_url`、`feature_vector`
- 状态字段：`status`、`is_deleted`
- 归属字段：`owner_id`

#### 3.2.4 状态设计

物品状态 `status` 采用以下枚举值：

- `pending`：待认领或待后续匹配
- `matched`：已形成匹配关系
- `closed`：已完成认领或已手动关闭

逻辑删除字段 `is_deleted` 用于区分正常展示与下架状态，避免物理删除造成历史链路丢失。

#### 3.2.5 业务规则

- 用户仅可编辑和管理本人发布的物品记录。
- 已下架物品可恢复重新上架。
- 已完成或已关闭物品不再参与正常公开展示与后续匹配。
- 图片存在时，系统优先自动识别类别并提取特征向量。

### 3.3 SYS3 图像识别与以图搜物模块

#### 3.3.1 模块职责

该模块对应需求规格说明书中的“图像识别与以图搜物模块”，负责图像预处理、类别识别、特征提取、相似检索及相似推荐。

#### 3.3.2 主要实现文件

- [search.py](F:/Last_offering/items/backend/app/api/v1/search.py)
- [ai_classifier.py](F:/Last_offering/items/backend/app/services/ai_classifier.py)

- [SearchView.vue](F:/Last_offering/items/frontend/src/views/SearchView.vue)

#### 3.3.3 图像处理流程

用户上传图片后，系统按以下流程处理：

1. 保存原始图片到媒体目录。
2. 对图片进行基础读取与预处理。
3. 调用分类模型输出类别预测结果。
4. 调用特征提取模型生成向量表示。
5. 将分类结果与特征向量写入数据库。
6. 在需要时触发图像检索或自动匹配。

#### 3.3.4 以图搜物设计

接口：`POST /api/v1/search/by-image`

输入参数主要包括：

- 图片文件
- 检索方向 `search_type`
- 返回数量 `top_k`
- 相似度阈值 `threshold`

输出内容主要包括：

- 查询图像的类别识别结果
- Top-K 类别建议
- 相似物品结果列表
- 每条候选结果的相似度

#### 3.3.5 相似推荐设计

系统在物品详情页提供“相似物品推荐”能力，通过当前物品的特征向量，在反向类型物品集合中执行相似检索并返回排序后的候选项。

- 接收自然语言寻物描述
- 自动提取类别、颜色、品牌、地点、关键词等结构化条件
- 对候选物品进行综合评分与排序
- 返回结果及评分原因说明

系统支持两种解析模式：

- 本地快速解析模式
- ARK 增强解析模式

该设计既保证响应速度，又兼顾解析质量与系统可用性。

该功能在前端页面中采用“先快速解析、再人工检查确认”的交互模式，以兼顾效率与准确性。

#### 3.4.5 语义归一化设计

为了提高召回率和匹配质量，系统对以下文本信息进行规范化处理：

- 类别同义词归一化
- 颜色同义词归一化
- 品牌别名与品牌家族合并
- 地点别名与层级映射
- 关键词清洗与标准化

该部分能力为智能寻物和自动匹配提供了统一语义基础。

### 3.6 匹配与认领流程模块

#### 3.6.1 模块职责

该模块用于建立失物与招领信息之间的关联关系，并完成从疑似匹配、人工确认到最终认领完成的全流程控制。

#### 3.6.2 主要实现文件

- [match.py](F:/Last_offering/items/backend/app/api/v1/match.py)
- [auto_match.py](F:/Last_offering/items/backend/app/services/auto_match.py)
- [ItemDetailView.vue](F:/Last_offering/items/frontend/src/views/ItemDetailView.vue)

#### 3.6.3 自动匹配设计

当新物品创建成功并具备图像特征向量后，后台自动启动匹配任务。自动匹配采用混合评分方法，综合考虑：

- 图像相似度
- 关键词重合程度
- 颜色一致性
- 品牌一致性或品牌家族关联性

系统为匹配结果生成原因说明，并在满足阈值条件时自动通知相关用户。

#### 3.6.4 手动匹配设计

用户在物品详情页点击“疑似遗失”后，可基于自己名下记录发起匹配申请。该流程弥补了自动匹配无法完全覆盖的场景，提高了系统实际可用性。

#### 3.6.5 匹配状态流转设计

匹配记录 `status` 定义如下：

- `pending`：已提交匹配申请，待确认
- `confirmed`：双方已建立有效匹配
- `rejected`：匹配被拒绝
- `completed`：双方确认线下交接完成

此外，系统维护以下确认字段：

- `lost_owner_confirmed`
- `found_owner_confirmed`

用于分别记录失主与拾得方是否确认完成交接。

#### 3.6.6 认领业务流程

完整认领流程如下：

1. 用户发布失物或招领信息。
2. 系统自动匹配或用户手动发起“疑似遗失”申请。
3. 另一方核对信息后确认或拒绝匹配。
4. 匹配确认后系统向双方推送联系方式。
5. 双方线下完成物品交接。
6. 双方分别在系统中点击完成认领。
7. 双方都确认后，匹配状态更新为 `completed`，相关物品状态更新为 `closed`。

## 7. AI 模块详细设计

### 7.1 图像智能设计

图像智能模块由图像分类、特征提取和相似检索三部分构成。

其设计思路为：

- 使用分类模型对物品图像进行类别预测。
- 使用特征提取模型输出高维向量表示。
- 通过余弦相似度完成图像检索与推荐。

该模块直接支撑需求规格说明书中的图像预处理、物品类别识别、图像特征提取和相似度检索等要求。

### 7.2 自动匹配设计

自动匹配模块在图像检索基础上引入了多因素融合评分。相比纯图像相似度检索，该模块更贴近真实失物招领场景，因为实际认领通常还需要颜色、品牌、地点和关键词等多维线索共同支撑。

### 7.3 语义智能设计

语义智能模块是当前系统相对早期需求的扩展能力，其核心价值在于：

- 降低用户录入门槛
- 提升弱结构化线索利用效率
- 让检索过程更接近自然语言习惯

系统通过本地规则保障可用性，再通过 ARK 增强模式提高解析质量，体现了“可落地”和“有创新”的平衡设计思路。

## 8. 安全设计与异常处理

### 8.1 安全设计

系统主要从以下方面保证安全性：

- 密码采用哈希加密存储
- 业务接口默认启用登录校验
- 管理接口单独进行管理员权限校验
- 物品删除采用逻辑删除策略
- 联系方式仅在匹配确认后向相关用户开放

### 8.2 异常处理设计

系统对常见异常场景进行了统一处理：

- 参数不合法时返回 400
- 未登录或令牌失效时返回 401
- 权限不足时返回 403
- 资源不存在时返回 404
- 服务器内部错误时返回 500

前端通过统一请求封装处理错误提示，并在必要时引导用户重新登录。

## 9. 部署设计

### 9.1 本地开发部署

本地开发时，系统通常采用如下模式运行：

- 后端通过 Uvicorn 启动，默认监听 `8000` 端口
- 前端通过 Vite 启动，默认监听 `5173` 端口
- MySQL 单独运行，供后端连接

### 9.2 Docker 部署

项目提供 Docker Compose 配置，用于快速部署：

- `mysql`：数据库服务
- `backend`：FastAPI 服务
- `frontend`：Nginx 静态站点与反向代理服务

部署时：

- 前端静态文件由 Nginx 提供服务
- `/api/` 与 `/media/` 请求由 Nginx 转发至后端
- 上传媒体目录通过卷挂载保存
